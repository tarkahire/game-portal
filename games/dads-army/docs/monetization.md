# Pay-to-Play Model

## Core Principle

**NO PAY-TO-WIN. COMPETITIVE INTEGRITY IS ABSOLUTE.**

No amount of money spent can give a player any gameplay advantage — no stronger units, no faster production, no extra resources, no additional map information, no diplomatic leverage. Every player on a server competes on equal mechanical footing regardless of spending.

This is not just a design guideline — it is the foundational promise of the game. Violating it would destroy the strategic depth that makes Dad's Army worth playing.

## Premium Subscription

A monthly subscription that provides quality-of-life and cosmetic features.

### Premium Features

| Feature | Details |
|---------|---------|
| Simultaneous servers | Join up to **3 active servers** at once (vs 1 for free) |
| Premium servers | Access to 2x speed servers and special map configurations |
| Competitive servers | Access to entry-fee servers with prize pools |
| Cosmetic customization | City visual themes, unit color schemes, custom army banners, map marker styles |
| Priority queue | When a server is full, premium players are placed higher in the waiting list |
| Profile badge | "Premium" badge visible to other players on your profile |
| Extended post-game | Battle history replay available for 90 days (vs 30 for free) |

### Pricing

| Plan | Price | Billing |
|------|-------|---------|
| Monthly | $4.99/month | Charged monthly, cancel anytime |
| Annual | $39.99/year ($3.33/month) | Charged annually, cancel anytime (no refund for remaining months) |

### Premium Edge Cases

- **Server overlap:** If a premium player downgrades to free while in 3 active servers, they continue in all 3 until each ends. They cannot join new servers until they are in 1 or fewer active servers.
- **Premium expiry mid-server:** The player continues in their current server(s) with no interruption. Premium cosmetics remain visible for the duration of that server. Queue priority and premium server access are lost immediately.
- **Re-subscribing:** All premium features activate immediately upon re-subscription. No waiting period.

## Server Entry Fee (Competitive Servers)

Premium-only servers that charge a small entry fee, creating a prize pool for top finishers.

### How It Works

1. Admin creates a Competitive server with a set entry fee.
2. Players pay the entry fee to join (deducted from account balance or charged directly).
3. All entry fees go into a prize pool.
4. When the server ends, the prize pool is distributed to the top 3 finishers.

### Prize Distribution

| Placement | Share | Example ($1.99 fee, 30 players = $59.70 pool) |
|-----------|-------|----------------------------------------------|
| 1st place | 50% | $29.85 |
| 2nd place | 30% | $17.91 |
| 3rd place | 20% | $11.94 |

Prizes are distributed as **account credit** (not cash withdrawal). Account credit can be used for future entry fees, premium subscription, or cosmetic purchases.

### Entry Fee Tiers

| Tier | Fee | Typical Pool (30 players) | Risk Level |
|------|-----|--------------------------|------------|
| Casual | $0.99 | $29.70 | Low |
| Standard | $1.99 | $59.70 | Medium |
| High Stakes | $2.99 | $89.70 | High |

### Revenue Model

The platform takes a **10% rake** from each prize pool before distribution. This is the platform's revenue from competitive servers.

Example: $59.70 pool → $5.97 to platform, $53.73 distributed to winners.

## Implementation

### Database Schema (Supabase)

**`user_profiles` table:**

| Column | Type | Description |
|--------|------|-------------|
| id | UUID (PK) | User ID (from Supabase Auth) |
| subscription_tier | ENUM('free', 'premium') | Current subscription status |
| subscription_expires | TIMESTAMP | When premium expires (null for free) |
| account_credit | DECIMAL | Prize pool winnings / credit balance |
| reputation | INTEGER | Cross-server reputation score (0-100) |
| created_at | TIMESTAMP | Account creation date |

**`user_cosmetics` table:**

| Column | Type | Description |
|--------|------|-------------|
| id | UUID (PK) | Row ID |
| user_id | UUID (FK → user_profiles) | Owner |
| cosmetic_id | VARCHAR | Identifier for the cosmetic item |
| cosmetic_type | ENUM('city_skin', 'unit_pack', 'battle_effect', 'profile_frame', 'name_color', 'banner', 'victory_animation') | Category |
| purchased_at | TIMESTAMP | Purchase date |
| is_equipped | BOOLEAN | Currently active/selected |

**`transactions` table:**

| Column | Type | Description |
|--------|------|-------------|
| id | UUID (PK) | Transaction ID |
| user_id | UUID (FK → user_profiles) | Buyer |
| type | ENUM('subscription', 'cosmetic', 'server_entry', 'prize_payout') | Transaction type |
| amount_cents | INTEGER | Amount in cents (positive = charge, negative = payout) |
| stripe_payment_id | VARCHAR | Stripe payment intent ID |
| status | ENUM('pending', 'completed', 'refunded', 'failed') | Transaction status |
| created_at | TIMESTAMP | Transaction date |

### Stripe Integration

- **Subscriptions:** Stripe Billing with monthly/annual recurring payment.
- **One-time purchases:** Stripe Checkout for cosmetic items.
- **Server entry fees:** Stripe Payment Intents.
- **Payouts (prize pools):** Account credit system — no direct cash withdrawals. Credit is stored in `user_profiles.account_credit` and deducted when used for purchases.

### Frontend Gating

Premium features are gated by checking `user_profiles.subscription_tier` in the frontend:

```javascript
// Example: checking if user can join a premium server
const canJoinPremiumServer = userProfile.subscription_tier === 'premium';

// Example: checking if user can join another server
const activeServers = await getActiveServersForUser(userId);
const maxServers = userProfile.subscription_tier === 'premium' ? 3 : 1;
const canJoinNewServer = activeServers.length < maxServers;

// Example: checking if user owns a cosmetic
const ownedCosmetics = await getUserCosmetics(userId);
const hasCitySkin = ownedCosmetics.some(c => c.cosmetic_id === 'city_skin_industrial');
```

**Important:** Cosmetic checks are purely frontend. The server does not validate cosmetics because they have zero gameplay impact. This reduces server load. Subscription tier and server access are validated server-side via Supabase RLS policies.

### RLS Policies

```sql
-- Users can only read their own profile
CREATE POLICY "Users read own profile"
ON user_profiles FOR SELECT
USING (auth.uid() = id);

-- Users can only read their own cosmetics
CREATE POLICY "Users read own cosmetics"
ON user_cosmetics FOR SELECT
USING (auth.uid() = user_id);

-- Only the server (service role) can insert/update transactions
CREATE POLICY "Service role manages transactions"
ON transactions FOR ALL
USING (auth.role() = 'service_role');

-- Users can read their own transactions
CREATE POLICY "Users read own transactions"
ON transactions FOR SELECT
USING (auth.uid() = user_id);
```

## Anti-Cheat Considerations

Because Dad's Army uses a **server-authoritative architecture** (Supabase handles all game state), client-side cheating is inherently limited:

| Threat | Mitigation |
|--------|------------|
| Modified client (stat hacking) | All combat, production, and resource calculations happen server-side. The client only displays results. A modified client cannot make units stronger. |
| Packet manipulation | Supabase RLS policies ensure players can only modify their own data. Attempts to write to other players' data are rejected. |
| Speed hacking | Tick-based system means all actions happen at server-defined intervals. A client cannot "speed up" their turns. |
| Vision hacking (fog of war) | Fog of war is enforced server-side. The client only receives data for hexes the player can see. Enemy unit positions in fogged hexes are not sent to the client. |
| Auto-play bots | Difficult to prevent entirely. Rate limiting on actions per tick mitigates. Suspicious patterns (perfectly optimized moves 24/7) flagged for review. |

### Multi-Accounting (Primary Cheat Risk)

The biggest cheat threat is **multi-accounting**: one human player running multiple accounts on the same server to gain an unfair advantage (e.g., using alt accounts to feed resources to a main account, or coordinating multiple armies under one person's control).

**Countermeasures:**

| Measure | Details |
|---------|---------|
| IP monitoring | Flag accounts that frequently share IP addresses and are on the same server. Not conclusive (shared households exist) but triggers review. |
| Device fingerprinting | Browser fingerprinting to detect same-device access across accounts. |
| Behavioral analysis | Detect accounts that always trade with each other, always ally, never fight each other, and have suspiciously coordinated movements. |
| Player reporting | Other players can report suspected multi-accounting. Reports are reviewed by admins. |
| Penalties | Confirmed multi-accounting: all accounts banned from the server. Repeat offenders: permanent account ban. |
| Phone verification | Optional but encouraged: verified accounts get a "Verified" badge. Competitive servers may require phone verification to enter. |

## Pricing Philosophy

- **Low barrier to entry:** Free tier is genuinely complete. Players should never feel forced to pay.
- **Reasonable premium:** $4.99/month is less than a coffee. The multi-server access alone justifies it for active players.
- **Cosmetics are optional luxuries:** No one needs them. They exist for players who want to personalize their experience.
- **No FOMO:** No limited-time offers, no seasonal exclusives, no "buy now or miss out" pressure. All items are permanently available.
- **No loot boxes:** Every purchase is deterministic. Players know exactly what they are getting before they pay. No randomized rewards, no gacha mechanics, no gambling-adjacent systems.
- **Competitive integrity above all:** If a monetization idea could even arguably affect gameplay balance, it is rejected. Zero tolerance.
